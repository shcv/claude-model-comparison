Headline finding: Opus 4.5 rewrites previously-edited regions 1.2× more often (17.7% vs 14.8%)—the strongest accuracy signal in the dataset. Opus 4.5 triggered 42 user-directed corrections; Opus 4.6 triggered none. Opus 4.6 achieves lower rework while costing 9–45% less per task despite producing 1.4× more output tokens.

The two models succeed at comparable rates but work in fundamentally different ways. Opus 4.6 uses 75% more tool calls, deploys subagents that are 94% read-only research, and plans more often. Of 115 statistical tests with Bonferroni correction, eight survive—mostly about how each model works (tool usage, distributions), not whether they succeed. Several initial findings were reversed during analysis when more direct signals became available (§11).

Opus 4.5 acts first and adjusts as needed; Opus 4.6 investigates first and implements in concentrated bursts. This report examines where these strategies diverge across 1,920 tasks (1,727 vs 193), ordered from the strongest signals (cost, accuracy) through behavioral mechanics to weaker observational patterns.

Classification

Each detected overlap is classified by context:

  • Self-correction: Agent rewrites its own output within the same task, no intervening error—catching and fixing its own mistakes unprompted.
  • Error recovery: Same task, but a tool error (test failure, syntax error, compilation failure) occurred between the original edit and the rewrite.
  • User-directed correction: The rewrite occurs in a later task, and the intervening user message contains dissatisfaction signals (“wrong”, “revert”, “fix that”, etc.).
  • Iterative refinement: Sequential edits to the same region without error signals or dissatisfaction—normal iterative development.

Opus 4.6 self-corrects at a higher rate than Opus 4.5 (55.6% vs 9.2% of overlapping edits), but has zero error recoveries (vs 15.1% for Opus 4.5). Opus 4.5 triggered 42 user-directed corrections (10.7% of overlaps); Opus 4.6 triggered none. The overall rewrite rate (17.7% vs 14.8%) is the strongest accuracy signal: Opus 4.5 rewrites previously-edited regions 1.2× as often as Opus 4.6.

Opus 4.5 rarely plans (1.4% overall). Opus 4.6 concentrates its planning on harder tasks: 42.9% of complex+ tasks and 3.8% of moderate tasks, with no planning for trivial or simple tasks. Neither model plans for trivial tasks.

Insufficient evidence that planning is necessary: Although Opus 4.6 is the only model that plans, this data alone cannot establish that planning causes its better outcomes. For complex+ tasks, planned tasks score higher (5.00 vs 3.00), but this is based on just 1 scored planned task vs 4 scored unplanned tasks. Furthermore, only 5 of Opus 4.6’s 133 tasks use planning at all, making any causal claim extremely fragile. The same disposition that leads 4.6 to plan also leads it to explore more, use more subagents, and investigate before acting; planning may be a symptom of thoroughness rather than its mechanism. A controlled experiment (forcing planning on/off) would be needed to isolate the effect.

10. Model Profiles

Opus 4.5 — The Executor

Strategy: Act first, adjust as needed. Jumps to implementation with minimal upfront research.

Subagents: 37% general-purpose (implementation workers), 49% Explore. Primarily user-directed (55%).

Planning: Rarely uses planning mode (1.0%). Distributes research evenly through the task.

Strength: Efficient on trivial and simple tasks. Lower tool overhead. Stable performance across session lengths.

Risk: Higher failure rate on complex tasks. May miss edge cases that upfront research would catch.

Opus 4.6 — The Explorer

Strategy: Research first, implement in focused bursts. Front-loads investigation before touching files.

Subagents: 94% Explore (read-only research), 0% general-purpose. Evenly split autonomous/user-directed (50%).

Planning: Occasionally self-initiates planning (4.0% of tasks, 4 of 99).

Strength: Higher completion rate (+3.7pp), lower failure rate (−5.5pp). Best in short, focused sessions.

Risk: Higher tool overhead (75% more calls). Performance degrades in long sessions (−0.57 alignment over 9+ tasks).

Each Claude Code session was segmented into tasks based on user-message boundaries and topic shifts. An LLM annotator classified each task for complexity, type, sentiment, completion status, and alignment score (1–5 scale). Behavioral metrics—subagent usage, planning, parallelization—were extracted from tool-call logs.

Edit timeline analysis tracked 2,948 Edit/Write tool calls across all sessions, building per-file content ownership maps to detect when a later edit’s old_string overlaps with an earlier edit’s new_string. Overlaps are detected via three tiers (exact match, containment at ≥30% ratio, and Jaccard line overlap) and classified by context (self-correction, error recovery, user-directed, iterative refinement). Edit metrics were joined with task classifications to compute complexity-binned accuracy rates (93% and 88% coverage for the two models).

Compaction analysis found 0 compact_boundary system messages across 0 compacting sessions, extracting trigger type, pre-compaction token count, and position. Outcome impact was measured by splitting tasks into pre/post groups at the first compaction timestamp, with a control group of non-compacting sessions split at the median compaction position to isolate position effects.

Statistical testing: 115 tests were conducted: chi-square (categorical), Mann-Whitney U (continuous), proportion tests (rates). Bonferroni-corrected threshold: p<0.00043. Eight findings survive correction: lines removed, lines per minute, tools per file, and five categorical distributions (task completion, scope management, iteration required, error recovery, communication quality). Significance notes appear inline in each section.

Development Process

This analysis was developed iteratively. Several approaches were tried and abandoned when they proved unreliable:

Automated dissatisfaction detection (abandoned as primary signal): Initial LLM-based sentiment classification flagged 7–9% dissatisfaction for both models. LLM-agent audit of all 59 flagged cases revealed 73–93% false positive rates—the classifiers were fooled by task-coordination language (e.g., “fix” in subagent prompts). This approach was replaced by edit timeline analysis, which detects self-corrections mechanistically rather than through sentiment proxies.

LLM quality judgement (dropped): An LLM judge was asked to compare code quality between models. The results were unreliable—the judge lacked sufficient context to evaluate whether code met domain requirements, and produced confident but ungrounded assessments. This section was removed entirely rather than presented with caveats.

Refactoring routing recommendation (revised): Early signals suggested Opus 4.5 was better for refactoring, based on four false-positive dissatisfaction flags and small sample sizes. Sentiment detection and git retention both proved unreliable. Edit timeline analysis provided a direct mechanistic signal: tracking every edit across all sessions revealed that Opus 4.5 rewrites previously-edited regions at 1.2× the rate of Opus 4.6 (63.9% vs 51.7%), reversing the initial recommendation.

Limitations

Single user: All data comes from one developer’s workflow. Results may not generalize to other users, codebases, or task distributions.

Observational, not experimental: Tasks were not randomly assigned to models. Opus 4.6 was used later chronologically and on different (often harder) tasks, confounding model effects with task effects.

Complexity confound: Opus 4.6’s different complexity mix (37.3% moderate-and-above vs 32.6% for Opus 4.5) inflates its resource usage metrics and may suppress its satisfaction scores relative to what it would achieve on equivalent tasks.

Sample asymmetry: The 8.9:1 ratio (1,727 vs 193 tasks) means Opus 4.5 estimates have narrower confidence intervals. Effect sizes for Opus 4.6 are less precise.

LLM-in-the-loop analysis: All task classification, sentiment analysis, and dissatisfaction auditing was performed by LLM agents (Claude Haiku and Sonnet), with human review of outputs and skepticism-driven re-auditing. No classification was done by hand.

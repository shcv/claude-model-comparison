Study design note: This is a single-user observational study with non-random task assignment. Opus 4.5 data spans 2 months across 31 projects (1,742 tasks); Opus 4.6 spans 5 days across 4 projects (202 tasks). Differences may reflect the user, project mix, or time period rather than model capabilities. All findings should be treated as hypotheses for future testing.

Observed patterns suggest the two models work differently at comparable success rates. Opus 4.6 uses 68% more tool calls, deploys subagents that are 94% read-only research, and plans more often. Of 115 statistical tests with Bonferroni correction, eight survive—mostly about how each model works (tool usage, distributions), not whether they succeed. Several initial findings were reversed during analysis when more direct signals became available (§11).

The observed data is consistent with a pattern where Opus 4.5 acts first and adjusts as needed while Opus 4.6 investigates first and implements in concentrated bursts—but the confounded study design means this characterization is tentative. This report examines where observed patterns diverge, ordered from the strongest signals (cost, tool usage) through behavioral mechanics to weaker observational patterns.

Normalizing by session hours rather than task count reinforces the per-task finding: Opus 4.5 costs $12.75/session-hour ($5,327 over 417.9h) vs $7.51/session-hour for Opus 4.6 ($781 over 104.0h)—a 41% reduction. Counter-intuitive cost finding (with caveats): Despite producing more output tokens, Opus 4.6 appears 12–47% cheaper per task from trivial through moderate levels, likely from more efficient prompt caching. The exception is major tasks, where 4.6 costs +63% more—but this is based on 5 4.6 major tasks vs 18 for 4.5, so the estimate is unreliable. These figures come from organic sessions where task mix, session structure, and user behavior differ between models. Without more controlled experiments, per-complexity cost advantages should be interpreted cautiously—some of the gap may reflect differences in how the models were used rather than inherent efficiency.

Classification

Each detected overlap is classified by context:

  • Self-correction: Agent rewrites its own output within the same task, no intervening error—catching and fixing its own mistakes unprompted.
  • Error recovery: Same task, but a tool error (test failure, syntax error, compilation failure) occurred between the original edit and the rewrite.
  • User-directed correction: The rewrite occurs in a later task, and the intervening user message contains dissatisfaction signals (“wrong”, “revert”, “fix that”, etc.).
  • Iterative refinement: Sequential edits to the same region without error signals or dissatisfaction—normal iterative development.

Opus 4.6 self-corrects at a higher rate than Opus 4.5 (55.6% vs 9.4% of overlapping edits), but has zero error recoveries (vs 15.0% for Opus 4.5). Opus 4.5 triggered 42 user-directed corrections (10.7% of overlaps); Opus 4.6 triggered none. The overall rewrite rate (17.1% vs 8.8%) is the strongest accuracy signal: Opus 4.5 rewrites previously-edited regions 1.9× as often as Opus 4.6.

Opus 4.5 rarely plans (1.9% overall). Opus 4.6 concentrates its planning on harder tasks: 63.6% of complex+ tasks and 7.4% of moderate tasks, with no planning for trivial or simple tasks. Neither model plans for trivial tasks.

Insufficient evidence that planning is necessary: Although Opus 4.6 is the only model that plans, this data alone cannot establish that planning causes its better outcomes. The same disposition that leads 4.6 to plan also leads it to explore more, use more subagents, and investigate before acting; planning may be a symptom of thoroughness rather than its mechanism. A controlled experiment (forcing planning on/off) would be needed to isolate the effect.

8. Context Window & Compaction

Claude Code compacts the conversation when the context window fills, summarizing earlier messages to free space. Community discussion has speculated that Opus 4.6’s higher token usage might lead to more frequent compaction. Compaction was extremely rare: Opus 4.5 triggered 1 compaction event in 1 of 231 sessions (0.4%), while Opus 4.6 had none across 21 sessions. With only a single event, compaction is essentially a non-factor in this dataset, though the one compacting session does provide a limited before/after comparison.

10. Model Profiles

Opus 4.5 — Observed Pattern

Observed approach: Tends to act first and adjust as needed. Jumps to implementation with minimal upfront research.

Subagents: 37% general-purpose (implementation workers), 49% Explore. Primarily autonomous (55%).

Planning: Rarely uses planning mode (1.9%). Distributes research evenly through the task.

Observed strengths: Efficient on trivial and simple tasks. Lower tool overhead. Stable performance across session lengths.

Observed weaknesses: Higher failure rate on complex tasks. Higher rewrite rate (17.1%) may indicate missed requirements.

Opus 4.6 — Observed Pattern

Observed approach: Tends to research first, then implement in focused bursts. Front-loads investigation before touching files.

Subagents: 94% Explore (read-only research), 0% general-purpose. Evenly split autonomous/user-directed (50%).

Planning: Occasionally self-initiates planning (8.7% of tasks, 9 of 104).

Observed strengths: Higher completion rate (+2.0pp), lower failure rate (−5.8pp). Lower rewrite rate (8.8%, n=29 tasks).

Observed weaknesses: Higher tool overhead (68% more calls). Performance may degrade in long sessions.

Each Claude Code session was segmented into tasks based on user-message boundaries and topic shifts. An LLM annotator classified each task for complexity, type, sentiment, completion status, and alignment score (1–5 scale). Behavioral metrics—subagent usage, planning, parallelization—were extracted from tool-call logs.

Edit timeline analysis tracked 3,208 Edit/Write tool calls across all sessions, building per-file content ownership maps to detect when a later edit’s old_string overlaps with an earlier edit’s new_string. Overlaps are detected via three tiers (exact match, containment at ≥30% ratio, and Jaccard line overlap) and classified by context (self-correction, error recovery, user-directed, iterative refinement). Edit metrics were joined with task classifications to compute complexity-binned accuracy rates (93% and 88% coverage for the two models).

Compaction analysis found 1 compact_boundary system message across 1 compacting session, extracting trigger type, pre-compaction token count, and position. Outcome impact was measured by splitting tasks into pre/post groups at the first compaction timestamp, with a control group of non-compacting sessions split at the median compaction position to isolate position effects.

Statistical testing: 115 tests were conducted: chi-square (categorical), Mann-Whitney U (continuous), proportion tests (rates). Bonferroni-corrected threshold: p<0.00043. Eight findings survive correction: tool calls, tools per file, and six categorical distributions (task completion, scope management, iteration required, error recovery, communication quality, autonomy level). Significance notes appear inline in each section.

Development Process

This analysis was developed iteratively. Several approaches were tried and abandoned when they proved unreliable:

Automated dissatisfaction detection (abandoned as primary signal): Initial LLM-based sentiment classification flagged 7–9% dissatisfaction for both models. LLM-agent audit of all 59 flagged cases revealed 73–93% false positive rates—the classifiers were fooled by task-coordination language (e.g., “fix” in subagent prompts). This approach was replaced by edit timeline analysis, which detects self-corrections mechanistically rather than through sentiment proxies.

LLM quality judgement (dropped): An LLM judge was asked to compare code quality between models. The results were unreliable—the judge lacked sufficient context to evaluate whether code met domain requirements, and produced confident but ungrounded assessments. This section was removed entirely rather than presented with caveats.

Refactoring routing recommendation (revised): Early signals suggested Opus 4.5 was better for refactoring, based on four false-positive dissatisfaction flags and small sample sizes. Sentiment detection and git retention both proved unreliable. Edit timeline analysis provided a more direct mechanistic signal: tracking every edit across all sessions revealed that Opus 4.5 rewrites previously-edited regions more often (17.1% vs 8.8%), though Opus 4.6’s small editing sample (n=29 tasks) means the confidence intervals overlap substantially.

Limitations

Single user: All data comes from one developer’s workflow. Results may not generalize to other users, codebases, or task distributions.

Observational, not experimental: Tasks were not randomly assigned to models. Opus 4.6 was used later chronologically and on different (often harder) tasks, confounding model effects with task effects.

Complexity confound: Opus 4.6’s different complexity mix (40.1% moderate-and-above vs 33.2% for Opus 4.5) inflates its resource usage metrics and may suppress its satisfaction scores relative to what it would achieve on equivalent tasks.

Sample asymmetry: The 8.6:1 ratio (1,742 vs 202 tasks) means Opus 4.5 estimates have narrower confidence intervals. Effect sizes for Opus 4.6 are less precise.

LLM-in-the-loop analysis: All task classification, sentiment analysis, and dissatisfaction auditing was performed by LLM agents (Claude Haiku and Sonnet), with human review of outputs and skepticism-driven re-auditing. No classification was done by hand.

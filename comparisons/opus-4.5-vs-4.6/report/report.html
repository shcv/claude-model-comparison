Headline finding: Users correct Opus 4.5 edits 16× more often (8.9% vs 0.6%)—the strongest accuracy signal in the dataset, holding across all complexity levels and task durations. Opus 4.6 achieves this while costing 9–45% less per task despite producing 2.7× more output tokens.

The two models succeed at comparable rates but work in fundamentally different ways. Opus 4.6 uses 55% more tool calls, deploys 3× more subagents (72% read-only research), and plans 10× more often. Of 105 statistical tests with Bonferroni correction, only three survive—all about how each model works (tool usage, alignment score), not whether they succeed. Several initial findings were reversed during analysis when more direct signals became available (§11).

Opus 4.5 acts first and adjusts as needed; Opus 4.6 investigates first and implements in concentrated bursts. This report examines where these strategies diverge across 2,395 tasks (2,035 vs 360), ordered from the strongest signals (cost, accuracy) through behavioral mechanics to weaker observational patterns.

Classification

Each detected overlap is classified by context:

  • Self-correction: Agent rewrites its own output within the same task, no intervening error—catching and fixing its own mistakes unprompted.
  • Error recovery: Same task, but a tool error (test failure, syntax error, compilation failure) occurred between the original edit and the rewrite.
  • User-directed correction: The rewrite occurs in a later task, and the intervening user message contains dissatisfaction signals (“wrong”, “revert”, “fix that”, etc.).
  • Iterative refinement: Sequential edits to the same region without error signals or dissatisfaction—normal iterative development.

Self-correction rates are comparable (4.5% vs 6.1%), meaning both models catch their own mistakes at similar rates. Error recovery is also comparable (~9–12%). The critical difference is user-directed corrections: users had to ask Opus 4.5 to fix its work 16× more often (8.9% vs 0.6% of edits). This is the strongest accuracy signal in the dataset—it measures how often the model produces output that the user explicitly rejects.

Iterative Refinement: Workflow or Strategy?

The aggregate iterative refinement gap (39.7% vs 12.6%) invites the question: does Opus 4.5 genuinely iterate more, or does the gap reflect different usage patterns? Breaking it down by complexity and duration reveals a striking reversal: at the complex+ and long-task end, Opus 4.6 actually iterates more (31.7% vs 25.0%), while Opus 4.5 dominates trivial and short-task bins (26.3% vs 2.7%). The aggregate gap is concentrated in exactly the categories where workflow differences—user participation, task granularity, self-selection into harder challenges—are strongest.

This does not undermine the user-correction signal: corrections require explicit user dissatisfaction regardless of workflow. But the iterative refinement gap is substantially a workflow artifact rather than a pure model-strategy difference.

Both models plan more on harder tasks, but Opus 4.6 plans at substantially higher rates at every complexity level. The gap is largest for moderate tasks (22.0% vs 3.8%). Neither model plans much for trivial tasks.

Insufficient evidence that planning is necessary: Although Opus 4.6 plans 9.2× more often, this data alone cannot establish that planning causes its better outcomes. Opus 4.6 achieves comparable alignment on the 79% of complex tasks where it does not plan (4.18 unplanned vs 4.12 planned). Furthermore, alignment deltas by complexity bin are inconsistent—positive for simple and moderate tasks in both models, but near-zero or negative for complex+ tasks—further undermining any causal claim. The same disposition that leads 4.6 to plan also leads it to explore more, use more subagents, and investigate before acting; planning may be a symptom of thoroughness rather than its mechanism. A controlled experiment (forcing planning on/off) would be needed to isolate the effect.

Both models compact at similar token thresholds (~155K) and positions (~60% through session).

10. Model Profiles

Opus 4.5 — The Executor

Strategy: Act first, adjust as needed. Jumps to implementation with minimal upfront research.

Subagents: 50% general-purpose (implementation workers), 32% Explore. Primarily user-directed (72%).

Planning: Rarely uses planning mode (1.7%). Distributes research evenly through the task.

Strength: Efficient on trivial and simple tasks. Lower tool overhead. Stable performance across session lengths.

Risk: Higher failure rate on complex tasks. May miss edge cases that upfront research would catch.

Opus 4.6 — The Explorer

Strategy: Research first, implement in focused bursts. Front-loads investigation before touching files.

Subagents: 72% Explore (read-only research), 16% general-purpose. Overwhelmingly autonomous (85.5%).

Planning: Frequently self-initiates planning (15.7% of tasks, 88% autonomous).

Strength: Higher completion rate (+6.2pp), lower failure rate (−5.8pp). Best in short, focused sessions.

Risk: Higher tool overhead (55% more calls). Performance degrades in long sessions (−0.57 alignment over 9+ tasks).

Each Claude Code session was segmented into tasks based on user-message boundaries and topic shifts. An LLM annotator classified each task for complexity, type, sentiment, completion status, and alignment score (1–5 scale). Behavioral metrics—subagent usage, planning, parallelization—were extracted from tool-call logs.

Edit timeline analysis tracked 3,600 Edit/Write tool calls across all sessions, building per-file content ownership maps to detect when a later edit’s old_string overlaps with an earlier edit’s new_string. Overlaps are detected via three tiers (exact match, containment at ≥30% ratio, and Jaccard line overlap) and classified by context (self-correction, error recovery, user-directed, iterative refinement). Edit metrics were joined with task classifications to compute complexity-binned accuracy rates (93% and 88% coverage for the two models).

Compaction analysis parsed 53 compact_boundary system messages across 34 compacting sessions, extracting trigger type, pre-compaction token count, and position. Outcome impact was measured by splitting tasks into pre/post groups at the first compaction timestamp, with a control group of non-compacting sessions split at the median compaction position to isolate position effects.

Statistical testing: 105 tests were conducted: chi-square (categorical), Mann-Whitney U (continuous), proportion tests (rates). Bonferroni-corrected threshold: p<0.00048. Only three findings survive correction: tool calls per task, tools per file, and alignment score. Significance notes appear inline in each section.

Development Process

This analysis was developed iteratively. Several approaches were tried and abandoned when they proved unreliable:

Automated dissatisfaction detection (abandoned as primary signal): Initial LLM-based sentiment classification flagged 7–9% dissatisfaction for both models. LLM-agent audit of all 59 flagged cases revealed 73–93% false positive rates—the classifiers were fooled by task-coordination language (e.g., “fix” in subagent prompts). This approach was replaced by edit timeline analysis, which detects self-corrections mechanistically rather than through sentiment proxies.

LLM quality judgement (dropped): An LLM judge was asked to compare code quality between models. The results were unreliable—the judge lacked sufficient context to evaluate whether code met domain requirements, and produced confident but ungrounded assessments. This section was removed entirely rather than presented with caveats.

Refactoring routing recommendation (revised): Early signals suggested Opus 4.5 was better for refactoring, based on four false-positive dissatisfaction flags and small sample sizes. Sentiment detection and git retention both proved unreliable. Edit timeline analysis provided a direct mechanistic signal: tracking every edit across all sessions revealed that users correct Opus 4.5 edits 16× more often (8.9% vs 0.6%), reversing the initial recommendation.

Limitations

Single user: All data comes from one developer’s workflow. Results may not generalize to other users, codebases, or task distributions.

Observational, not experimental: Tasks were not randomly assigned to models. Opus 4.6 was used later chronologically and on different (often harder) tasks, confounding model effects with task effects.

Complexity confound: Opus 4.6’s higher proportion of complex tasks (11.9% vs 8.2%) inflates its resource usage metrics and may suppress its satisfaction scores relative to what it would achieve on equivalent tasks.

Sample asymmetry: The 5.6:1 ratio (2,035 vs 360 tasks) means Opus 4.5 estimates have narrower confidence intervals. Effect sizes for Opus 4.6 are less precise.

LLM-in-the-loop analysis: All task classification, sentiment analysis, and dissatisfaction auditing was performed by LLM agents (Claude Haiku and Sonnet), with human review of outputs and skepticism-driven re-auditing. No classification was done by hand.

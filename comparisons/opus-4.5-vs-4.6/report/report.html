What Changes When You Switch

Across 529 statistical tests, 93 survive Bonferroni correction—and most describe how the model works, not whether it succeeds. The overall success rates are comparable; what changes is the experience of working alongside it. These are the five most noticeable differences, drawn from one user’s workflow over 2,320 tasks.

1. The model plans before it acts—and you stop steering it

Opus 4.6 uses formal planning mode on 13.9% of tasks vs 1.7% for 4.5, rising to 51% at complex and 72% at major difficulty. It front-loads codebase investigation with a 2.2× longer explore phase, deploying subagents that are 74% read-only researchers (vs 47% for 4.5). The practical effect is a shift from interactive collaboration to delegation: you issue a prompt and return to find completed work rather than course-correcting mid-task. This shows up in the data as fewer user-directed corrections across all complexity levels (§4, §8).

2. It thinks less often, but more carefully

The single largest effect across all 529 tests is thinking fraction (d=0.67, §3). Opus 4.5 activates extended thinking on 75% of requests regardless of difficulty; 4.6 activates on 58% but averages 4,175 characters when it does (vs 2,635). On trivial tasks, 4.6 often skips thinking entirely. On complex tasks, it thinks deeply. This calibration means compute is allocated where it matters rather than spread uniformly across every interaction.

3. Fewer rewrites, better first-attempt accuracy

Opus 4.6 rewrites its own edits 10.3% of the time vs 16.6%—a 38% reduction. Its self-correction rate is actually higher (3.3% vs 1.2%), meaning it catches its own mistakes rather than having the user point them out. Failure rates drop from 13.2% to 8.6%, and alignment scores improve significantly (p=0.000023, one of 15 overall Bonferroni survivors). The “plan first” approach appears to pay off in execution accuracy (§5, §6).

4. Sessions get longer—you trust it with more

Median task duration rises 60% (67s vs 42s), with fewer ultra-short interactions (34% of tasks under 30 seconds vs 42%). The task mix shifts toward moderate-to-complex work issued in a single instruction, and 4.6 runs more tasks in the background for parallel execution. This isn’t purely a model capability difference—it’s a workflow adaptation. When the model handles larger tasks reliably, the user gives it larger tasks, waits longer, and intervenes less. The 8× increase in planning mode (§4) and 53% more tool calls per task (§7) are partly a consequence of this delegation shift.

5. Cost stays flat where it counts

Despite 2.4× more output tokens and 53% more tool calls per task, 4.6 is 10–35% cheaper at trivial through moderate complexity—the bulk of daily work. The reason is counterintuitive: output tokens are less than 7% of per-task cost, while cache operations account for ~95%. Opus 4.6 writes 24% less to cache (the most expensive token category at $18.75/MTok), more than offsetting its higher output and cache reads. Cost only tips higher at 30+ API requests, where cumulative cache reads compound past the write savings. Overall per-task cost is $2.62 vs $2.46: functionally neutral for a meaningfully different style of work (§2).

The data is consistent with a tentative characterization: Opus 4.5 acts first and adjusts, while Opus 4.6 investigates first and implements in concentrated bursts. The confounded study design means this framing is a hypothesis, not a conclusion. The analysis was iterative—several initial findings were revised or reversed when more direct signals became available (§10).

How This Report Works

This report is itself a Claude Code project. The analysis pipeline, statistical tests, table generation, and report assembly are all automated Python scripts, most of which were written with substantial assistance from Opus 4.6—the same model being evaluated. This recursive quality is worth noting both as an interesting artifact and as context for assessing the results.

Data Source

All data comes from one user’s real Claude Code sessions during the Early Access Program—not synthetic benchmarks or controlled experiments. Session logs are JSONL files containing every API request, tool call, and response. The dataset spans 463 sessions across 35 projects, representing $7,411 in API costs over two months of daily software engineering work.

Automated Pipeline

A 10-step pipeline transforms raw session logs into the finished report: session collection, task extraction (segmenting at user-message boundaries), regex and LLM classification (complexity, type, quality, sentiment), behavioral analysis, token extraction, statistical testing, and report generation. Each step tracks input/output file hashes and skips re-computation when inputs haven’t changed. The LLM classifier (Haiku) is verified against a regex baseline with 100% unknown-task resolution and 55% agreement on classified tasks.

Report Generation

Every number in the report is computed, not typed. Template variables ({{section.metric}}) resolve directly from analysis JSON files. Named table markers pull column definitions from spec files and generate HTML from data. An LLM prose-check pass verifies that narrative text is consistent with current numbers. 529 statistical tests use Bonferroni correction—the most conservative standard—to minimize false positives given the observational design. The Methodology section provides full technical detail on every step.

What to Trust

All numbers, tables, and statistical results are computed deterministically from session data and are reproducible by re-running the pipeline. Interpretive prose was drafted and checked with LLM assistance and may contain errors or overstatements. Effect sizes and p-values are exact; narrative claims linking those numbers to causal explanations are hypotheses, not conclusions. The sensitivity analysis validates key findings against restricted datasets excluding shared projects.

The cost advantage reverses at complex and major tiers, where accumulated cache reads over many requests outweigh the write savings.

Why Output Doesn’t Drive Cost

The 2.4× output difference seems like it should dominate the cost comparison, but output tokens are a minor cost component. Cache operations dwarf everything else:

Three forces offset to produce the $0.16 net difference. 4.6 writes 24% fewer tokens to cache per task (51K vs 68K), saving $0.31 at the most expensive category ($18.75/M—10× the read price). But 4.6 reads 34% more cached context (794K vs 594K), costing $0.38 at the cheapest category ($1.875/M). And the 2.4× output increase costs only $0.10. The write savings nearly cancel the read and output increases: −$0.31 + $0.38 + $0.10 = +$0.17.

Why does 4.6 write less to cache? Per-request analysis reveals two mechanisms. First, 4.6 starts tasks with a leaner cache footprint—its first API request writes 49% fewer cache tokens than 4.5’s, suggesting a more compact or reusable context structure. Second, when cache cools (gap >5 minutes between tasks), both models experience cold starts at identical rates (~25% of transitions), but 4.5 must re-write 177K tokens to repopulate vs 87K for 4.6. Cold cache causes a 6.4× write inflation for 4.5 but only 2.1× for 4.6. The likely explanation: 4.6’s “investigate then execute” pattern creates more compact, reusable context, while 4.5’s incremental approach builds a larger accumulated context that costs more to re-cache after cooling.

This mechanism produces the complexity-dependent curve above. At trivial through moderate tiers (typically ≤30 API requests), lean initialization dominates and 4.6 is 9–35% cheaper. At 30+ requests, cumulative cache reads compound past the initialization savings—4.6’s per-request cache reads are slightly higher (89K vs 80K in the 30+ tier), and spread over 50+ requests, this adds up.

Each Claude Code session was segmented into tasks based on user-message boundaries and topic shifts. An LLM annotator classified each task for complexity, type, sentiment, completion status, and alignment score (1–5 scale). Behavioral metrics—subagent usage, planning, parallelization—were extracted from tool-call logs.

Minimum sample sizes: Cross-cut cells with fewer than 5–10 observations (depending on test type) are excluded from statistical testing. This primarily affects the “major” complexity tier (n=19/18) and rare task types.

Edit timeline analysis tracked 4,799 Edit/Write tool calls across all sessions, building per-file content ownership maps to detect when a later edit’s old_string overlaps with an earlier edit’s new_string. Overlaps are detected via three tiers (exact match, containment at ≥30% ratio, and Jaccard line overlap) and classified by context (self-correction, error recovery, user-directed, iterative refinement). Edit metrics were joined with task classifications to compute complexity-binned accuracy rates (93% and 88% coverage for the two models).

Compaction analysis found 79 compact_boundary system messages across 50 compacting sessions, extracting trigger type, pre-compaction token count, and position. Outcome impact was measured by splitting tasks into pre/post groups at the first compaction timestamp, with a control group of non-compacting sessions split at the median compaction position to isolate position effects.

Statistical testing: 529 tests were conducted: chi-square (categorical), Mann-Whitney U (continuous), proportion tests (rates). Bonferroni correction was chosen over less conservative alternatives (e.g., Benjamini-Hochberg FDR) to minimize false positives given the observational design; the tradeoff is higher false-negative risk. Corrected threshold: p<0.0000945. Across all 529 tests (overall and per-complexity strata), 93 findings survive Bonferroni and 168 survive FDR correction. At the overall level, fifteen survive Bonferroni, including alignment score (p=0.000023), duration (p<0.000001, though d=0.005—negligible practical effect), tool calls/task (p<0.000001, d=−0.29), tools/file (p<0.000001, d=−0.10), and three categorical distributions (task completion, communication quality, autonomy level). All three chi-square survivors have low-expected-cell-count warnings, which may inflate test statistics. Significance notes appear inline in each section.

Development Process

This analysis was developed iteratively. Several approaches were tried and abandoned when they proved unreliable:

Automated dissatisfaction detection (abandoned as primary signal): Initial LLM-based sentiment classification flagged 7–9% dissatisfaction for both models. LLM-agent audit of all 59 flagged cases revealed 73–93% false positive rates—the classifiers were fooled by task-coordination language (e.g., “fix” in subagent prompts). This approach was replaced by edit timeline analysis, which detects self-corrections mechanistically rather than through sentiment proxies.

LLM quality judgement (dropped): An LLM judge was asked to compare code quality between models. The results were unreliable—the judge lacked sufficient context to evaluate whether code met domain requirements, and produced confident but ungrounded assessments. This section was removed entirely rather than presented with caveats.

Refactoring routing recommendation (revised): Early signals suggested Opus 4.5 was better for refactoring, based on four false-positive dissatisfaction flags and small sample sizes. Sentiment detection and git retention both proved unreliable. Edit timeline analysis provided a more direct mechanistic signal: tracking every edit across all sessions revealed Opus 4.5 with a 16.6% rewrite rate vs 10.3% for Opus 4.6 (now 246 editing tasks, 120 overlapping edits).

Limitations

LLM-in-the-loop analysis: All task classification, sentiment analysis, and alignment scoring was performed by LLM agents (Claude Haiku and Sonnet). This creates a circularity concern: Claude models are classifying Claude model outputs. No formal inter-rater reliability was computed. Human spot-checks validated flagged cases, but systematic bias between models (e.g., if the classifier is more generous toward outputs that resemble its own style) cannot be ruled out. All three overall chi-square Bonferroni survivors and the alignment score depend on LLM-generated categories.

Single user: All data comes from one developer’s workflow. Results may not generalize to other users, codebases, or task distributions.

Temporal confound: Opus 4.5 spans 2 months; Opus 4.6 spans 8 days. A productive week, a particular project focus, or simply the novelty of a new model could color all 591 Opus 4.6 tasks simultaneously. The null hypothesis—that all observed differences reflect the user’s changing work patterns rather than model capabilities—cannot be rejected by this design.

Observational, not experimental: Tasks were not randomly assigned to models. Opus 4.6 was used later chronologically and on different (often harder) tasks, confounding model effects with task effects.

Complexity confound: Opus 4.6’s different complexity mix (42% moderate-and-above vs 33% for Opus 4.5) inflates its resource usage metrics and may suppress its satisfaction scores. Complexity-stratified comparisons would isolate model effects from mix effects but are not yet in the pipeline.

Platform evolution: The Claude Code SDK evolved between December 2025 and February 2026. Changes to system prompts, available tools, or subagent defaults could contribute to behavioral differences attributed to the models.

Sample asymmetry: The 2.9:1 ratio (1,729 vs 591 tasks) means Opus 4.5 estimates have narrower confidence intervals. Effect sizes for Opus 4.6 are less precise.

User learning effect: The user may have learned to use Claude Code more effectively over time, benefiting whichever model came second in the chronological sequence.

Study design note: This is a single-user observational study with non-random task assignment. Opus 4.5 data spans 2 months across 31 projects (1,751 tasks); Opus 4.6 spans 5 days across 8 projects (272 tasks). Differences may reflect the user, project mix, or time period rather than model capabilities. All findings should be treated as hypotheses for future testing.

Observed patterns suggest the two models work differently at comparable success rates. Opus 4.6 uses 51% more tool calls, deploys subagents that are 69% read-only Explore agents, and plans more often. Of 115 statistical tests with Bonferroni correction, six survive—mostly about how each model works (tool usage, distributions), not whether they succeed. Several initial findings were reversed during analysis when more direct signals became available (§7).

The observed data is consistent with a pattern where Opus 4.5 acts first and adjusts as needed while Opus 4.6 investigates first and implements in concentrated bursts—but the confounded study design means this characterization is tentative. This report examines where observed patterns diverge, ordered from the strongest signals (cost, tool usage) through behavioral mechanics. Areas with insufficient sample sizes are deferred to §6.

Despite producing more output tokens and making more API round-trips (9.4 vs 7.4 requests/task), Opus 4.6 shows lower per-task costs at trivial through moderate levels, likely reflecting higher cache hit rates from concentrated project usage during the shorter 4.6 evaluation period.

Normalizing by session hours rather than task count reinforces the per-task finding: Opus 4.5 costs $12.73/session-hour ($5,285 over 415.1h) vs $8.66/session-hour for Opus 4.6 ($998 over 115.2h)—a 32% reduction. Session hours measure wall-clock time from first to last message, so this metric includes idle time and is not a direct measure of active coding cost. Still, it confirms the per-task cost advantage extends to session-level efficiency.

Counter-intuitive cost finding (with caveats): Despite producing more output tokens, Opus 4.6 appears 17–52% cheaper per task from trivial through moderate levels, likely from more efficient prompt caching. The exception is major tasks, where 4.6 costs +63% more—but this is based on 5 4.6 major tasks vs 18 for 4.5, so the estimate is unreliable. These figures come from organic sessions where task mix, session structure, and user behavior differ between models. Without more controlled experiments, per-complexity cost advantages should be interpreted cautiously—some of the gap may reflect differences in how the models were used rather than inherent efficiency.

5. Model Profiles

Opus 4.5 — Observed Pattern

Observed approach: Tends to act first and adjust as needed. Jumps to implementation with minimal upfront research.

Subagents: 47% Explore, 37% general-purpose (implementation workers). Primarily autonomous (54%).

Planning: Rarely uses planning mode (1.7%). Distributes research evenly through the task.

Observed strengths: Efficient on trivial and simple tasks. Lower tool overhead. Stable performance across session lengths.

Observed weaknesses: Higher rewrite rate (16.6%). More tool calls go to implementation vs research.

Opus 4.6 — Observed Pattern

Observed approach: Tends to research first, then implement in focused bursts. Front-loads investigation before touching files.

Subagents: 69% Explore (read-only research), 25% general-purpose. More autonomous (66%).

Planning: Uses planning mode more often (6.8% of tasks, 19 of 279).

Observed strengths: Lower rewrite rate (15.6%). Lower cost at trivial–moderate complexity.

Observed weaknesses: Higher tool overhead (51% more calls). Costlier at major complexity (n=5).

6. Future Investigations

The following areas showed suggestive patterns but lacked the sample sizes needed for reliable conclusions. The analysis pipeline continues to collect data for each; they will be promoted to full sections when readiness thresholds are met. Run scripts/check_readiness.py --dir comparisons/opus-4.5-vs-4.6 to check current status.

Edit Accuracy

Edit timeline analysis detects when a model rewrites its own earlier output. Current data shows Opus 4.5 with a 16.6% rewrite rate vs 15.6% for Opus 4.6, with 89 editing tasks and 59 overlapping edits for 4.6. The gap has narrowed substantially with more data, making the previous directional finding unreliable until sample sizes grow further.

Promotion threshold: ≥50 editing tasks and ≥30 overlapping edits for 4.6. Currently: 89 tasks, 59 overlaps. Both thresholds met; pending meaningful rate divergence.

Quality & Satisfaction

LLM-annotated sentiment and completion distributions are nearly identical between models. All comparisons are statistical ties after Bonferroni correction (closest: alignment p=0.002, threshold p<0.00043).

Promotion threshold: Any quality metric surviving Bonferroni correction (p<0.00043). Currently: closest p=0.002.

Planning

Opus 4.6 enters plan mode on 6.8% of tasks (19 of 279) vs 1.7% for Opus 4.5 (29 of 1,728). The alignment benefit is suggestive but fragile: only 19 planned tasks for 4.6, making any causal inference unreliable.

Promotion threshold: ≥30 planned tasks for 4.6. Currently: 19.

Compaction

Context-window compaction occurs in 10% of 4.5 sessions (33/326, 52 events) and 16% of 4.6 sessions (9/58, 16 events). The 68 total events now exceed the promotion threshold; pending analysis of pre/post outcome differences.

Promotion threshold: ≥10 total compaction events across both models. Currently: 68. Threshold met.

Session Dynamics

Warm-up effects, effort distribution, and session-length patterns show observational differences but no significance testing survives Bonferroni correction. Duration test is p=0.07, well above the corrected threshold.

Promotion threshold: Any session-level test surviving Bonferroni correction. Currently: none.

Each Claude Code session was segmented into tasks based on user-message boundaries and topic shifts. An LLM annotator classified each task for complexity, type, sentiment, completion status, and alignment score (1–5 scale). Behavioral metrics—subagent usage, planning, parallelization—were extracted from tool-call logs.

Edit timeline analysis tracked 3,735 Edit/Write tool calls across all sessions, building per-file content ownership maps to detect when a later edit’s old_string overlaps with an earlier edit’s new_string. Overlaps are detected via three tiers (exact match, containment at ≥30% ratio, and Jaccard line overlap) and classified by context (self-correction, error recovery, user-directed, iterative refinement). Edit metrics were joined with task classifications to compute complexity-binned accuracy rates (93% and 88% coverage for the two models).

Compaction analysis found 68 compact_boundary system messages across 42 compacting sessions, extracting trigger type, pre-compaction token count, and position. Outcome impact was measured by splitting tasks into pre/post groups at the first compaction timestamp, with a control group of non-compacting sessions split at the median compaction position to isolate position effects.

Statistical testing: 115 tests were conducted: chi-square (categorical), Mann-Whitney U (continuous), proportion tests (rates). Bonferroni-corrected threshold: p<0.00043. Six findings survive correction: tool calls/task (p=0.0004), tools/file (p=0.0002), lines/minute (p=0.0004), and three categorical distributions (communication quality, scope management, task completion). Significance notes appear inline in each section.

Development Process

This analysis was developed iteratively. Several approaches were tried and abandoned when they proved unreliable:

Automated dissatisfaction detection (abandoned as primary signal): Initial LLM-based sentiment classification flagged 7–9% dissatisfaction for both models. LLM-agent audit of all 59 flagged cases revealed 73–93% false positive rates—the classifiers were fooled by task-coordination language (e.g., “fix” in subagent prompts). This approach was replaced by edit timeline analysis, which detects self-corrections mechanistically rather than through sentiment proxies.

LLM quality judgement (dropped): An LLM judge was asked to compare code quality between models. The results were unreliable—the judge lacked sufficient context to evaluate whether code met domain requirements, and produced confident but ungrounded assessments. This section was removed entirely rather than presented with caveats.

Refactoring routing recommendation (revised): Early signals suggested Opus 4.5 was better for refactoring, based on four false-positive dissatisfaction flags and small sample sizes. Sentiment detection and git retention both proved unreliable. Edit timeline analysis provided a more direct mechanistic signal: tracking every edit across all sessions revealed Opus 4.5 with a 16.6% rewrite rate vs 15.6% for Opus 4.6, a gap that has narrowed considerably as the Opus 4.6 sample has grown (now 89 editing tasks, 59 overlapping edits).

Limitations

Single user: All data comes from one developer’s workflow. Results may not generalize to other users, codebases, or task distributions.

Observational, not experimental: Tasks were not randomly assigned to models. Opus 4.6 was used later chronologically and on different (often harder) tasks, confounding model effects with task effects.

Complexity confound: Opus 4.6’s different complexity mix (39% moderate-and-above vs 33% for Opus 4.5) inflates its resource usage metrics and may suppress its satisfaction scores relative to what it would achieve on equivalent tasks.

Sample asymmetry: The 6.4:1 ratio (1,751 vs 272 tasks) means Opus 4.5 estimates have narrower confidence intervals. Effect sizes for Opus 4.6 are less precise.

LLM-in-the-loop analysis: All task classification, sentiment analysis, and dissatisfaction auditing was performed by LLM agents (Claude Haiku and Sonnet), with human review of outputs and skepticism-driven re-auditing. No classification was done by hand.

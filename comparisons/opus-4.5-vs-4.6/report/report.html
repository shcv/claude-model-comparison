Headline finding: Opus 4.5 rewrites previously-edited regions 1.2× more often (19.4% vs 15.9%)—the strongest accuracy signal in the dataset. Opus 4.5 triggered 69 user-directed corrections; Opus 4.6 triggered none. Opus 4.6 achieves lower rework while costing 9–45% less per task despite producing 1.3× more output tokens.

The two models succeed at comparable rates but work in fundamentally different ways. Opus 4.6 uses 180% more tool calls, deploys subagents that are 95% read-only research, and plans more often. Of 115 statistical tests with Bonferroni correction, only three survive—all about how each model works (tool usage, alignment score), not whether they succeed. Several initial findings were reversed during analysis when more direct signals became available (§11).

Opus 4.5 acts first and adjusts as needed; Opus 4.6 investigates first and implements in concentrated bursts. This report examines where these strategies diverge across 2,394 tasks (2,034 vs 360), ordered from the strongest signals (cost, accuracy) through behavioral mechanics to weaker observational patterns.

Classification

Each detected overlap is classified by context:

  • Self-correction: Agent rewrites its own output within the same task, no intervening error—catching and fixing its own mistakes unprompted.
  • Error recovery: Same task, but a tool error (test failure, syntax error, compilation failure) occurred between the original edit and the rewrite.
  • User-directed correction: The rewrite occurs in a later task, and the intervening user message contains dissatisfaction signals (“wrong”, “revert”, “fix that”, etc.).
  • Iterative refinement: Sequential edits to the same region without error signals or dissatisfaction—normal iterative development.

Opus 4.6 self-corrects at a higher rate than Opus 4.5 (26.1% vs 7.6% of overlapping edits), and shows a higher error recovery rate (21.7% vs 15.2%). Opus 4.5 triggered 69 user-directed corrections (12.8% of overlaps); Opus 4.6 triggered none. The overall rewrite rate (19.4% vs 15.9%) is the strongest accuracy signal: Opus 4.5 rewrites previously-edited regions 1.2× as often as Opus 4.6.

10. Model Profiles

Opus 4.5 — The Executor

Strategy: Act first, adjust as needed. Jumps to implementation with minimal upfront research.

Subagents: 40% general-purpose (implementation workers), 39% Explore. Primarily user-directed (57%).

Planning: Rarely uses planning mode (1.4%). Distributes research evenly through the task.

Strength: Efficient on trivial and simple tasks. Lower tool overhead. Stable performance across session lengths.

Risk: Higher failure rate on complex tasks. May miss edge cases that upfront research would catch.

Opus 4.6 — The Explorer

Strategy: Research first, implement in focused bursts. Front-loads investigation before touching files.

Subagents: 95% Explore (read-only research), 0% general-purpose. Overwhelmingly autonomous (52.6%).

Planning: Occasionally self-initiates planning (3.8% of tasks, 5 of 133).

Strength: Higher completion rate (+6.2pp), lower failure rate (−5.8pp). Best in short, focused sessions.

Risk: Higher tool overhead (55% more calls). Performance degrades in long sessions (−0.57 alignment over 9+ tasks).

Each Claude Code session was segmented into tasks based on user-message boundaries and topic shifts. An LLM annotator classified each task for complexity, type, sentiment, completion status, and alignment score (1–5 scale). Behavioral metrics—subagent usage, planning, parallelization—were extracted from tool-call logs.

Edit timeline analysis tracked 3,858 Edit/Write tool calls across all sessions, building per-file content ownership maps to detect when a later edit’s old_string overlaps with an earlier edit’s new_string. Overlaps are detected via three tiers (exact match, containment at ≥30% ratio, and Jaccard line overlap) and classified by context (self-correction, error recovery, user-directed, iterative refinement). Edit metrics were joined with task classifications to compute complexity-binned accuracy rates (93% and 88% coverage for the two models).

Compaction analysis parsed 49 compact_boundary system messages across 30 compacting sessions, extracting trigger type, pre-compaction token count, and position. Outcome impact was measured by splitting tasks into pre/post groups at the first compaction timestamp, with a control group of non-compacting sessions split at the median compaction position to isolate position effects.

Statistical testing: 115 tests were conducted: chi-square (categorical), Mann-Whitney U (continuous), proportion tests (rates). Bonferroni-corrected threshold: p<0.00043. Only three findings survive correction: tool calls per task, tools per file, and alignment score. Significance notes appear inline in each section.

Development Process

This analysis was developed iteratively. Several approaches were tried and abandoned when they proved unreliable:

Automated dissatisfaction detection (abandoned as primary signal): Initial LLM-based sentiment classification flagged 7–9% dissatisfaction for both models. LLM-agent audit of all 59 flagged cases revealed 73–93% false positive rates—the classifiers were fooled by task-coordination language (e.g., “fix” in subagent prompts). This approach was replaced by edit timeline analysis, which detects self-corrections mechanistically rather than through sentiment proxies.

LLM quality judgement (dropped): An LLM judge was asked to compare code quality between models. The results were unreliable—the judge lacked sufficient context to evaluate whether code met domain requirements, and produced confident but ungrounded assessments. This section was removed entirely rather than presented with caveats.

Refactoring routing recommendation (revised): Early signals suggested Opus 4.5 was better for refactoring, based on four false-positive dissatisfaction flags and small sample sizes. Sentiment detection and git retention both proved unreliable. Edit timeline analysis provided a direct mechanistic signal: tracking every edit across all sessions revealed that Opus 4.5 rewrites previously-edited regions at 1.2× the rate of Opus 4.6 (63.9% vs 51.7%), reversing the initial recommendation.

Limitations

Single user: All data comes from one developer’s workflow. Results may not generalize to other users, codebases, or task distributions.

Observational, not experimental: Tasks were not randomly assigned to models. Opus 4.6 was used later chronologically and on different (often harder) tasks, confounding model effects with task effects.

Complexity confound: Opus 4.6’s higher proportion of complex tasks (9.8% vs 9.7%) inflates its resource usage metrics and may suppress its satisfaction scores relative to what it would achieve on equivalent tasks.

Sample asymmetry: The 5.7:1 ratio (2,034 vs 360 tasks) means Opus 4.5 estimates have narrower confidence intervals. Effect sizes for Opus 4.6 are less precise.

LLM-in-the-loop analysis: All task classification, sentiment analysis, and dissatisfaction auditing was performed by LLM agents (Claude Haiku and Sonnet), with human review of outputs and skepticism-driven re-auditing. No classification was done by hand.

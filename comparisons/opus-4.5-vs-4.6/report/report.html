Study design note: This is a single-user observational study with non-random task assignment. Opus 4.5 data spans 2 months across 30 projects (1,729 tasks); Opus 4.6 spans 8 days across 14 projects (573 tasks). Differences may reflect the user, project mix, or time period rather than model capabilities. All findings should be treated as hypotheses for future testing.

Observed patterns suggest the two models work differently at comparable success rates. Opus 4.6 uses 58% more tool calls, deploys subagents that are 74% read-only Explore agents, and plans 8× more often. Of 115 statistical tests with Bonferroni correction, nine survive—mostly about how each model works (tool usage, distributions), not whether they succeed. Several initial findings were reversed during analysis when more direct signals became available (§9).

The observed data is consistent with a pattern where Opus 4.5 acts first and adjusts as needed while Opus 4.6 investigates first and implements in concentrated bursts—but the confounded study design means this characterization is tentative. This report examines where observed patterns diverge, ordered from the strongest signals (cost, tool usage) through behavioral mechanics and quality outcomes.

Normalizing by session hours rather than task count reinforces the per-task finding: Opus 4.5 costs $12.75/session-hour ($5,292 over 415.2h) vs $6.75/session-hour for Opus 4.6 ($2,055 over 304.5h)—a 47% reduction. Session hours measure wall-clock time from first to last message, so this metric includes idle time and is not a direct measure of active coding cost. Still, it confirms the per-task cost advantage extends to session-level efficiency.

Counter-intuitive cost finding (with caveats): Despite producing more output tokens, Opus 4.6 appears 10–35% cheaper per task from trivial through moderate levels, likely from more efficient prompt caching. The exception is major tasks, where 4.6 costs +29% more—but this is based on 18 4.6 major tasks vs 19 for 4.5, so the estimate is unreliable. These figures come from organic sessions where task mix, session structure, and user behavior differ between models. Without more controlled experiments, per-complexity cost advantages should be interpreted cautiously—some of the gap may reflect differences in how the models were used rather than inherent efficiency.

LLM-annotated alignment scores (1–5 scale) show Opus 4.6 scoring higher on average, an effect that survives Bonferroni correction (p=0.000001, d=-0.24). The failed rate difference is also significant: 8.2% of 4.6 tasks fail vs 14.1% for 4.5 (p=0.000). Three categorical distributions—task completion, iteration required, and communication quality—also survive Bonferroni, indicating the models differ in how they reach outcomes, not just in outcome rates.

Quality confound: Opus 4.6’s different complexity mix (42% moderate-and-above vs 33% for 4.5) means it tackles harder work on average. Despite this, it achieves higher alignment scores and a lower failure rate—suggesting genuine capability improvement, though task selection remains confounded.

Edit timeline analysis tracks every Edit and Write tool call, building per-file content ownership maps to detect when a model later overwrites its own earlier output. Opus 4.5 rewrites 16.6% of its edits vs 10.5% for Opus 4.6—a gap that has widened as the dataset grew. Overlap classification reveals the rewrites are predominantly iterative refinement (63% for 4.5, largest category), not error recovery.

Interpretation: A lower rewrite rate is consistent with Opus 4.6’s research-first approach—investigating before editing reduces the need for later corrections. However, some rework reflects healthy iteration rather than mistakes; the distinction between “self-correction” and “iterative refinement” is heuristic-based.

7. Model Profiles

Opus 4.5 — Observed Pattern

Observed approach: Tends to act first and adjust as needed. Jumps to implementation with minimal upfront research.

Subagents: 47% Explore, 37% general-purpose (implementation workers). Primarily autonomous (54%).

Planning: Rarely uses planning mode (1.7%). Distributes research evenly through the task.

Observed strengths: Efficient on trivial and simple tasks. Lower tool overhead. Stable performance across session lengths.

Observed weaknesses: Higher rewrite rate (16.6%). More tool calls go to implementation vs research.

Opus 4.6 — Observed Pattern

Observed approach: Tends to research first, then implement in focused bursts. Front-loads investigation before touching files.

Subagents: 74% Explore (read-only research), 15% general-purpose. More autonomous (84%).

Planning: Uses planning mode more often (14.1% of tasks, 81 of 573).

Observed strengths: Lower rewrite rate (10.5%). Lower cost at trivial–moderate complexity.

Observed weaknesses: Higher tool overhead (58% more calls). Costlier at major complexity (n=18).

8. Future Investigations

The following areas show suggestive patterns but need deeper analysis before drawing conclusions. The data thresholds are met; what remains is causal or structural analysis that the current pipeline cannot automate. Run scripts/check_readiness.py --dir comparisons/opus-4.5-vs-4.6 to check current status.

Planning

Opus 4.6 enters plan mode on 14.1% of tasks (81 of 573) vs 1.7% for Opus 4.5. Adoption scales steeply with complexity: 51.4% at complex, 72.2% at major. Planned tasks show a modest alignment benefit (+0.16 overall) that reverses at complex and major tiers, suggesting planning helps most at moderate complexity.

Next step: Planning is not randomly assigned—the model selects when to plan. Propensity-score matching or complexity-controlled analysis would help separate planning efficacy from task selection bias.

Session Dynamics

Task duration survives Bonferroni correction (p=0.000001). Opus 4.6 takes longer per task (median 129.6s vs 94.0s), with the explore phase running 2.2× longer at median (67.8s vs 31.3s). Effort distribution shows 4.6 allocates more tool calls to research (35.1% vs 28.3%) and fewer to implementation (17.5% vs 27.0%). Active-time cost is $24.66/hour for 4.6 vs $20.96/hour for 4.5 (5-min idle threshold).

Next step: Session overlap analysis (1,765 overlapping pairs, max concurrency 11) complicates per-session cost attribution. Active-time cost varies with idle threshold (see sensitivity table above), but the directional relationship is stable across all thresholds tested.

Compaction

Context-window compaction occurs in 10.1% of 4.5 sessions (33/327) and 12.9% of 4.6 sessions (17/132). Pre/post comparisons show improvement after compaction, but a position-adjusted control group reveals the effect is driven by session position, not compaction itself (position-adjusted effect: −0.13 for 4.5, −0.27 for 4.6).

Next step: Compaction appears to preserve rather than degrade performance. The slightly negative position-adjusted effect for 4.6 warrants monitoring. Pre/post outcome analysis on task completion and satisfaction (not just alignment) may reveal effects masked by single-metric analysis.

Each Claude Code session was segmented into tasks based on user-message boundaries and topic shifts. An LLM annotator classified each task for complexity, type, sentiment, completion status, and alignment score (1–5 scale). Behavioral metrics—subagent usage, planning, parallelization—were extracted from tool-call logs.

Edit timeline analysis tracked 4,774 Edit/Write tool calls across all sessions, building per-file content ownership maps to detect when a later edit’s old_string overlaps with an earlier edit’s new_string. Overlaps are detected via three tiers (exact match, containment at ≥30% ratio, and Jaccard line overlap) and classified by context (self-correction, error recovery, user-directed, iterative refinement). Edit metrics were joined with task classifications to compute complexity-binned accuracy rates (93% and 88% coverage for the two models).

Compaction analysis found 79 compact_boundary system messages across 50 compacting sessions, extracting trigger type, pre-compaction token count, and position. Outcome impact was measured by splitting tasks into pre/post groups at the first compaction timestamp, with a control group of non-compacting sessions split at the median compaction position to isolate position effects.

Statistical testing: 115 tests were conducted: chi-square (categorical), Mann-Whitney U (continuous), proportion tests (rates). Bonferroni-corrected threshold: p<0.00043. Nine findings survive correction: alignment score (p=0.000001), duration (p=0.000001), tool calls/task (p<0.000001), tools/file (p<0.000001), failed rate (p=0.00025), and four categorical distributions (task completion, iteration required, communication quality, autonomy level). Significance notes appear inline in each section.

Development Process

This analysis was developed iteratively. Several approaches were tried and abandoned when they proved unreliable:

Automated dissatisfaction detection (abandoned as primary signal): Initial LLM-based sentiment classification flagged 7–9% dissatisfaction for both models. LLM-agent audit of all 59 flagged cases revealed 73–93% false positive rates—the classifiers were fooled by task-coordination language (e.g., “fix” in subagent prompts). This approach was replaced by edit timeline analysis, which detects self-corrections mechanistically rather than through sentiment proxies.

LLM quality judgement (dropped): An LLM judge was asked to compare code quality between models. The results were unreliable—the judge lacked sufficient context to evaluate whether code met domain requirements, and produced confident but ungrounded assessments. This section was removed entirely rather than presented with caveats.

Refactoring routing recommendation (revised): Early signals suggested Opus 4.5 was better for refactoring, based on four false-positive dissatisfaction flags and small sample sizes. Sentiment detection and git retention both proved unreliable. Edit timeline analysis provided a more direct mechanistic signal: tracking every edit across all sessions revealed Opus 4.5 with a 16.6% rewrite rate vs 10.5% for Opus 4.6, a gap that has widened as the Opus 4.6 sample has grown (now 241 editing tasks, 120 overlapping edits).

Limitations

Single user: All data comes from one developer’s workflow. Results may not generalize to other users, codebases, or task distributions.

Observational, not experimental: Tasks were not randomly assigned to models. Opus 4.6 was used later chronologically and on different (often harder) tasks, confounding model effects with task effects.

Complexity confound: Opus 4.6’s different complexity mix (42% moderate-and-above vs 33% for Opus 4.5) inflates its resource usage metrics and may suppress its satisfaction scores relative to what it would achieve on equivalent tasks.

Sample asymmetry: The 3.0:1 ratio (1,729 vs 573 tasks) means Opus 4.5 estimates have narrower confidence intervals. Effect sizes for Opus 4.6 are less precise.

LLM-in-the-loop analysis: All task classification, sentiment analysis, and dissatisfaction auditing was performed by LLM agents (Claude Haiku and Sonnet), with human review of outputs and skepticism-driven re-auditing. No classification was done by hand.
